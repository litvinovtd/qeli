# Нагрузочное тестирование qeli

Данные собраны **2026-06-06** на 2-VM лаборатории, бинарь release (LTO=fat,
strip, panic=abort), версия **0.5.6**, SHA-256 `cf6d98b2cf28118a…`.
Сырые результаты — [release/benchmark_results.json](../release/benchmark_results.json),
оркестратор — [scripts/benchmark.py](../scripts/benchmark.py).

> Релиз **0.6.0** — рефакторинг (общий C#-слой, .NET 10, чистка); протокол, крипто и
> дата-плоскость **не менялись**, поэтому замеры 0.5.6 остаются актуальными для 0.6.0.

## Стенд

| параметр | значение |
|---|---|
| Сервер | `10.66.116.10`, Debian 13, kernel 6.12.88, 2 vCPU (QEMU, AES-NI есть), NIC virtio |
| Клиент | `10.66.116.11`, идентичный, та же L2-сеть |
| iperf3 | 3.18 |
| TUN MTU | 1400 во всех режимах |
| Шифр / KEX | ChaCha20-Poly1305 / X25519 + HKDF-SHA256 (reality-tls: внешний TLS — AES-128-GCM) |

**Методология.** Для каждого режима поднимаем **отдельный туннель** (flat-INI
конфиг сервера + клиента), `ping -c 20 -i 0.2`, затем `iperf3 -t 12` в обе
стороны для TCP, или развёртка `iperf3 -u -b {100..500}M -l 1200` для UDP. CPU
снимаем через `iperf3 cpu_utilization_percent` (host=iperf-клиент, remote=iperf-сервер,
% от 2 vCPU) **и** сэмплинг процесса `qeli` на сервере.

> ⚠️ Числа имеют разброс ±3–5% между прогонами (virtio-VM). Для пристинных цифр
> гонять на свежеперезагруженной лабе. Этот прогон — без обрывов, `ping` 0% во
> всех режимах.

> 🔴 **ВАЖНО про колонки `qeli CPU` ниже.** Они сняты `ps %cpu` = среднее за ВСЮ
> жизнь процесса → **сильно недооценивают** (показывают ~34%). Точный delta-замер
> по `/proc/<pid>/stat` за окно нагрузки ([scripts/multicore_probe.py](../scripts/multicore_probe.py))
> даёт **~150% = ~1.5 ядра** на ОДИН туннель: дата-плоскость **многоядерная**
> (decrypt-reader + encrypt-writer + TUN-pump — независимые tokio-задачи на разных
> ядрах), а не «одно ядро». Колонки `qeli CPU` в таблицах — лишь относительное
> сравнение режимов между собой, НЕ абсолютная загрузка ядра.

## TCP — все режимы

`up` = клиент→сервер (decrypt на сервере), `down` = сервер→клиент (`iperf3 -R`,
decrypt на клиенте). `qeli CPU` — busiest qeli-процесс на сервере (% от ОДНОГО
ядра, среднее/пик за прогон up). `RSS` — резидентная память worker'а.

| режим | ↑ up Mbps | ↓ down Mbps | RTT avg | retr ↑/↓ | qeli CPU ↑ avg/пик | RSS |
|---|---:|---:|---:|---:|---:|---:|
| **tcp-plain** (raw, без обфускации) | 560.6 | 707.4 | 1.69 ms | 320 / 880 | 34.1% / 58.5% | 7.4 MB |
| **tcp-fake-tls** (мимикрия под TLS) | 560.4 | 690.1 | 1.78 ms | 243 / 812 | 32.8% / 57.2% | 7.4 MB |
| **tcp-padding** (+ random padding) | 560.8 | 676.4 | 1.62 ms | 201 / 117 | 33.5% / 58.4% | 7.3 MB |
| **tcp-frag** (+ fragmentation) | 544.3 | 701.5 | 1.74 ms | 389 / 445 | 33.1% / 57.9% | 7.7 MB |
| **tcp-obfs** (ChaCha20 stream + WS-fronting) | 490.9 | 577.3 | 1.64 ms | 845 / 540 | 33.8% / 58.6% | 7.3 MB |
| **tcp-reality** (proxy-bridge, fake-TLS) | **571.3** | **716.7** | 1.69 ms | 250 / 1298 | 34.2% / 59.1% | 7.6 MB |
| **tcp-reality-tls** (настоящий TLS 1.3) | 515.1 | **318.6** | 1.98 ms | 1043 / 495 | 32.3% / 56.1% | 8.2 MB |

**Чтение:**
- **`plain` ≈ `fake-tls`** (560 vs 560 ↑; 707 vs 690 ↓): голый туннель и fake-TLS
  дают одинаковую скорость — fake-TLS handshake разовый, а разница во фрейминге
  (3 байта/пакет) в шуме. Снятие обфускации throughput **не поднимает** (упор — в AEAD).
- **Обфускация почти бесплатна** на fake-tls: padding/fragmentation/heartbeat в
  пределах шума.
- **`obfs` ~−12% ↑ / −16% ↓** (491/577 vs 560/690): цена ChaCha20-XOR поверх всего
  потока (двойное шифрование).
- **`reality` (proxy) ≈ `fake-tls`**: peek-and-decide один раз на TCP accept.
- **`reality-tls` — цена настоящего TLS, особенно на download** (515 ↑ / **319 ↓**).
  См. разбор ниже.
- Дата-плоскость **многоядерная** (delta-замер `/proc/stat`, не `ps`): один туннель
  ~1.5 ядра, нагрузка от многих клиентов раскладывается по ядрам. С версии с
  **`tun.queues`** (multi-queue TUN, дефолт auto=nproc — см. [CONFIG.md](CONFIG.md))
  параллелится и сама TUN-помпа — контролируемый A/B даёт **+18% агрегата** при 2 туннелях,
  single-flow без изменений (подробно — раздел [Multi-queue TUN](#multi-queue-tun-tunqueues--ab)
  ниже). Память — **~7–8 MB** RSS. RTT overhead ≈ **1.6–1.9 ms**.

## UDP — все режимы (развёртка по битрейту, % потерь)

| режим | 100M | 200M | 300M | 400M | 500M | устойчивая полоса |
|---|---:|---:|---:|---:|---:|---|
| **udp-faketls** (fake-tls, база UDP) | 0% | 0% | 0.34% | 0.5% | 6.3% | **~400 Mbps** (<0.5% до 400) |
| **udp-padding** | 0% | 0% | 0.12% | 0.97% | 7.27% | ~400 Mbps |
| **udp-quic** (masking) | 0% | 0.01% | 0.23% | 0.74% | 8.06% | ~400 Mbps |

UDP-плоскость чистая (<0.35%) до 300 Mbps, держит ~400 при <1% потерь,
насыщается около 500 (6–8% потерь). `plain`/`obfs`/`reality*` — только TCP.

### Почему нет `plain` на UDP (by design)

`plain` (raw) — TCP-only, и это сознательное решение, а не пробел в тестах
(сервер отвергает `mode = plain` + `transport = udp` на старте, см.
`server/mod.rs`). Две причины:

1. **Скорости не дало бы.** Узкое место — AEAD (ChaCha20-Poly1305), а не
   5-байтовый fake-TLS-заголовок на датаграмму. Тот же вывод, что и для TCP
   (`plain ≈ fake-tls`): снятие обёртки throughput не поднимает.
2. **Хуже для обхода блокировок.** Raw-UDP — это высокоэнтропийный поток без
   структуры, т.е. ровно сигнатура «fully encrypted traffic», по которой DPI
   (GFW/ТСПУ) и режет UDP. `udp-faketls` наоборот даёт прикрытие (датаграммы
   выглядят как TLS-записи). То есть raw-UDP был бы **red-flag**, а не маскировка.

Поэтому для UDP базовый режим — `fake-tls` (строки выше), и именно он
бенчмаркается как «UDP без доп. обфускации».

## reality-tls download: почему ~320 и что с этим (разобрано на лабе)

Туннель в `reality-tls` идёт **внутри** настоящего TLS 1.3 (rustls на сервере,
hand-rolled `realtls` на клиенте). На **скачивании** клиентский reader снимает
**два слоя последовательно**: внешний TLS-record (AES-128-GCM) **и** внутренний
qeli-record (ChaCha20-Poly1305), с двойным фреймингом — в одной задаче. Это
примерно половинит download относительно одно-слойных режимов (~700 → ~320).

Что проверено замером (диагностика на лабе):
- **AES-NI на VM есть** (не software-AES).
- Клиентский процесс `qeli` при reality-tls download пикует **~67% одного ядра**
  (не 100%) — т.е. это **не** чистый CPU-потолок, а комбинация двойного AEAD +
  двойного фрейминга + await-оверхеда в одном reader-таске.

**Что НЕ помогло:** оптимизация клиентского read-пути `RealTlsStream::poll_read`
(батч-дешифровка всех готовых записей за один poll, 64-КиБ read-буфер взамен
4-КиБ, курсор вместо per-record `drain`+аллокации) — корректная и оставлена в
коде (161 тест зелёный), но download не сдвинула (317 → 322 → 309 → 319 Mbps — в
пределах шума), т.к. узкое место **не** в буферизации/syscall'ах.

**Реальные направления (follow-up, design-изменения):**
1. **Убрать избыточный внутренний AEAD в reality-tls** — внешний TLS уже шифрует и
   аутентифицирует, внутренний ChaCha20 на дата-плоскости дублирует это. Внутри
   reality-tls можно гнать дату в `plain`/Raw-фрейминге (без второго AEAD), оставив
   qeli-handshake/auth — это ~вдвое снижает работу клиента на download. Решение с
   security-tradeoff (defence-in-depth), требует явного согласования.
2. **Распараллелить два крипто-слоя** по задачам/ядрам (TLS-decrypt в одном, inner
   в другом).

Для большинства сценариев ~320 Mbps download у reality-tls достаточно; это плата
за DPI-устойчивость уровня «настоящий TLS на проводе» (закрывает теллы 1.1–1.6,
см. [DPI-AUDIT.md](DPI-AUDIT.md)).

## Multi-queue TUN (`tun.queues`) — A/B

**Как работает.** `tun.queues` (per-profile, дефолт `0` = auto = `nproc`) задаёт, сколько
`IFF_MULTI_QUEUE`-очередей открывается на одном TUN-устройстве. Дата-плоскость качает их N
независимыми reader/forwarder/writer-задачами (tokio), а **ядро RSS-раскладывает исходящие
TUN-пакеты по очередям по flow**. Так параллелится сама TUN-помпа (раньше — единая воронка
reader+forwarder+writer ~1.5 ядра) в дополнение к уже многоядерному per-connection AEAD.
`1` = прежняя одно-поточная помпа (для отката). UDP параллелится через N воркеров на
`SO_REUSEPORT`-сокетах. На проводе ничего не меняется, клиентов пересобирать не нужно
(TUN — локальный интерфейс ядра ОС).

**Замер.** Контролируемый A/B на 2-ядерной лабе: 2 туннеля в отдельных netns, одновременный
upload, CPU воркера сервера — delta по `/proc/<pid>/stat` (может превышать 100%), хост — delta
по `/proc/stat`. Оркестратор — [scripts/multitunnel_probe.py](../scripts/multitunnel_probe.py)
(`QELI_TUN_QUEUES=1|2`).

| | `queues=1` (legacy) | `queues=2` (multi-queue) | Δ |
|---|---:|---:|---:|
| **1 туннель**, агрегат | 458 Mbps | 455 Mbps | ≈0% |
| **2 туннеля**, агрегат | 607 Mbps | **718 Mbps** | **+18%** |
| qeli CPU @2 тун. (% от одного ядра) | 159% | 167% | берёт больше ядра |
| потоков в воркере | 9 | 11 | +2 queue-ридера |
| server-host @2 тун. (% от всех ядер) | 93% | 95% | насыщен |

**Чтение:**
- **Один поток от очередей не ускоряется** (458 ≈ 455 Mbps): одиночный TCP упирается в свою
  per-connection decrypt-задачу (~1 ядро), а не в TUN-помпу — N очередям нечего раскладывать.
  Поэтому посегментный бенч выше (один туннель на режим) прироста от multi-queue и **не показывает
  — это by design, не регресс**.
- **Агрегат от многих туннелей ускоряется**: 607 → 718 Mbps (**+18%**); qeli при этом подтягивает
  больше ядра (159→167%) и поднимает +2 потока-ридера на очереди.
- **+18% — нижняя граница**: оба 2-туннельных прогона упёрлись в насыщение хоста (93–95% всех
  ядер), т.к. `iperf3`-сток крутится на ТОМ ЖЕ 2-ядерном сервере и съедает свободные ядра. На
  боевом сервере с удалённым стоком и бОльшим числом ядер выигрыш растёт с числом ядер/клиентов.

**Вывод:** `tun.queues=auto` — бесплатный дефолт (single-flow без изменений, агрегат под нагрузкой
быстрее); `=1` ставить только для отладки/отката.

## Итоговая сводка

| | TCP | UDP |
|---|---|---|
| Практический потолок (2 vCPU) | **~560–571 ↑ / ~690–717 ↓ Mbps** (plain/fake-tls/reality) | **~400 Mbps** при <1% потерь |
| Latency overhead | ~1.6–2.0 ms | ~1.6–1.9 ms |
| Память (worker RSS) | ~7–8 MB | ~7–8 MB |
| Стоимость обфускации fake-tls | ≈0 | small |
| Стоимость `obfs` | −12% ↑ / −16% ↓ | (obfs только TCP) |
| Стоимость `reality` (proxy) | ≈0 | — |
| Стоимость `reality-tls` | −8% ↑, **−54% ↓** (вложенный TLS) | — |
| `plain` (raw) | ≈ fake-tls | n/a (TCP-only) |

Самый быстрый и дешёвый по CPU — `plain`/`fake-tls`; за DPI-устойчивость платят
`obfs` (умеренно) и `reality-tls` (заметно на скачивании).
