## L1 — 🟡 Lockout-DoS по username → заменён на adaptive tarpit (✅)

**Файлы:** `qeli/src/server/mod.rs` (`FailedAuthTracker`), `qeli/src/server/handler.rs`
(`verify_client_auth`), `qeli/src/web/api/login.rs`, `qeli/src/web/auth.rs`.

**Проблема.** Старый `FailedAuthTracker` после `max_attempts` фейлов **жёстко лочил
сам username** (бакет `by_user` с `lockout-until`), проверяемый в `check()` независимо
от source-IP. Зная валидный логин, атакующий с любых IP делал 5 неудачных попыток и
**выбивал легитимного юзера** (классический account-lockout DoS). В прошлом аудите
(L1, 2026-06-10) оставлен «by design» — теперь закрыт без компромисса.

**Подход (внесено) — Вариант «adaptive tarpit».** Жёсткий лок остаётся **только по
source-IP** (`by_ip`); username **никогда не блокируется**. Вместо лока username под
активным перебором получает **ограниченную сверху экспоненциальную задержку**
(`user_tarpit()`: base 200мс → ×2 за каждый лишний фейл, потолок 3с). Caller спит её
перед Argon2-проверкой, поэтому:
- легит-юзер с **верным** паролем никогда не получает отказ — в худшем случае во время
  атаки его успешная попытка задерживается ≤3с (с **любого** IP, включая новый);
- распределённый перебор (ботнет, ротация IP) зарезан по агрегатной скорости угадывания;
- одиночный source-IP по-прежнему жёстко лочится `by_ip`.

API: `check(user,ip)` → `check_ip(ip)` + `user_tarpit(user)`; добавлен
`record_ip_failure(ip)` — путь отказа по server-key-proof пишет **только** в `by_ip`,
чтобы безключевой сканер не поднимал чужой username-tarpit. Применено единообразно и
в веб-панели (форма логина + Basic-auth).

**Приёмка:** флуд логина с множества IP даёт `user_tarpit(...) > 0`, но чистый IP
жертвы всегда `check_ip(...).is_ok()` — кодировано тестом
`username_flood_never_hard_blocks_a_clean_ip` (`mod.rs`).

## T1 — ⚪ Android: сравнение auth-proof не constant-time (✅)

**Файл:** `qeli-android/.../QeliService.kt` (`verifyServerAuth`).

**Проблема.** Proof сервера сверялся через `ByteArray.contentEquals()`, который
коротко-замыкается на первом несовпавшем байте → формально тайминг-оракул.
*Severity «critical» завышен*: `expected` выводится из эфемерных значений рукопожатия
(transcript/ephemeral) и **меняется каждое соединение**, так что итеративный
побайтовый перебор по тайменгу непрактичен. Чинится дёшево, как гигиена.

**Подход (внесено).** `MessageDigest.isEqual(receivedProof, expected)` (constant-time).
Сравнение pinned-ключа (`serverStaticPub.contentEquals(pinnedBytes)`) оставлено — это
публичный ключ, не секрет. Rust (`ct_eq`) и C# (`FixedTimeEquals`) уже были
constant-time — находка чисто Android.

## T6 — 🟡 Poisoned-lock рушит живую сессию (✅)

**Файл:** `qeli/src/server/handler.rs` (`SessionShared`).

**Проблема.** `stream_count()` (`unwrap_or(0)`) и `remove_stream()` (`Err(_) => true`)
на отравленном мьютексе `streams` сообщали «0 потоков» / «сессия пуста» → ложный
teardown живой сессии.

**Подход (внесено).** Все методы `SessionShared` (`pick_stream`, `kick_all`,
`remove_stream`, `stream_count` и новый `try_add_stream`) переведены на уже имеющийся
хелпер `lock_or_recover` — он восстанавливает guard из poison вместо тихой деградации.

## T7 — 🟡 TOCTOU: превышение `max_clients` (✅)

**Файл:** `qeli/src/server/handler.rs` (AUTH-ветка).

**Проблема.** Лимит проверялся под read-локом, а вставка сессии — отдельно под
write-локом; N одновременных connect могли все пройти проверку и превысить лимит.

**Подход (внесено).** Авторитетная перепроверка `by_ip.len() >= max_clients` **под тем
же write-локом**, что и вставка; при превышении — освобождение уже выданного IP
(`pool.release(&dkey)`) и отказ. Ранний read-чек оставлен как дешёвый fast-path
(не аллоцировать IP в типичном over-limit случае).

---

## T8 — 🟡 TOCTOU: превышение `max_streams` (✅)

**Файл:** `qeli/src/server/handler.rs` (`SessionShared`, `run_stream`).

**Проблема.** JOIN проверял `stream_count() >= max_streams`, а реальный `add_stream`
шёл позже отдельным захватом лока — параллельные JOIN проскакивали лимит.

**Подход (внесено).** Атомарный `try_add_stream()` — проверка длины и push под одним
локом; возвращает `false` при заполнении. В `run_stream` проигравший гонку поток
отбрасывается (лог + return) вместо превышения cap. Ранний чек в JOIN оставлен как
fast-path (быстрый отказ до отправки `JOINOK`).

---

## T9 — ⚪ Гонка `DeviceId()` (✅)

**Файл:** `qeli-shared/.../VpnTunnelBase.cs`.

**Проблема.** Без файлового лока два одновременных вызова (праймери + bonded-потоки на
старте) могли сгенерировать и записать два разных device-id.

**Подход (внесено).** Static-кэш `_deviceId` + `lock (_deviceIdLock)` — id
вычисляется один раз на процесс, read/generate/write сериализованы.

---

## T10 — 🟡 Утечка сокета при ошибке connect/handshake (✅)

**Файлы:** `qeli-shared/.../VpnTunnelBase.cs` (`OpenBondedStream`),
`qeli-android/.../QeliService.kt` (`openBondedStream`).

**Проблема.** Сокет создавался, и если `connect`/JOIN-handshake бросал **до**
добавления в список teardown — он не закрывался (утечка FD при флапающей сети,
особенно при повторных bonded-попытках).

**Подход (внесено).** Тело обёрнуто в `try/catch`: на любом исключении сокет
закрывается (`Close()`/`ch.close()`) и снимается из `_bondedSockets`/`bondedSockets`,
если уже был добавлен. Затем исключение пробрасывается дальше.

## Отложено / триаж

- **T2** (⬜) `73× expect()` / `291× unwrap()`. Сырой счётчик точен, но раздут
  тест-модулями и `lock().unwrap()`. Не «починить 364 места», а разовый триаж только
  тех, что достижимы из **разбора входящих байтов** (`try_into`/slice-индексация/
  `from_slice`/`parse` на wire-данных) в `protocol/`, `server/handler.rs`,
  `udp_handler.rs` → заменить на `?`/`ok_or`.
- **T11** (⬜) `Thread.sleep` в IO-ридерах Android. Каналы `configureBlocking(true)`,
  так что ветка `n==0`+`sleep` фактически мёртвая; таймауты лучше отдать `soTimeout`.
  Низкий приоритет, не бандлится с фиксами безопасности.

## Ложные находки / без действий

- **T4** (❎ by design) disable-user ставит `enabled=false` в памяти до `save`. Это
  fail-safe: юзер отключён в живом процессе даже при сбое записи, а `save_err`
  возвращается админу. Порядок оставлен.
- **T3** (❎ ложная) config.rs «lock-drop-then-use TOCTOU». Путь клонируется под локом
  (`p.clone()`), затем `drop` — это корректный snapshot, запись идёт по локальной
  копии; check-then-use по разделяемому состоянию отсутствует.
- **T5** (❎ недостижимо) `build_auth_ok` сериализует `serde_json::json!`-значение
  `Value`, которое практически не фейлится; `unwrap_or_default()` — мёртвая ветка.
- **T12** (❎ доверенный вход) `build_routes_json` форматирует `cidr/gateway/metric`
  из **админской серверной конфигурации**, не из ввода атакующего по сети. Не инъекция.
- **B3** (❎ устарело) «релиз не в VCS» — репозиторий под git (история коммитов, `release/`
  в индексе).
