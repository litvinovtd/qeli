## 1. Что должно совпадать у клиента и сервера

Почти ничего из этого **не согласуется на проводе** — обе стороны берут значения из
своих конфигов независимо. Поэтому расхождение почти всегда выглядит не как понятная
ошибка «параметры не совпали», а как обрыв на середине подключения.

Самый коварный случай — **`bind_static` (H-1)**: расходятся не проверки, а соли KDF,
поэтому хендшейк формально проходит, а первая же зашифрованная запись не расшифровывается.

| Что | Где у сервера | Где у клиента | Симптом при расхождении |
|---|---|---|---|
| **`bind_static` (H-1)** | `auth.bind_static_to_session` (**по умолчанию `true`**) | `bind_static` (по умолчанию `true`) | Соли KDF расходятся → стороны выводят **разные ключи**. Явной ошибки «параметры не совпали» нет: на сервере `handshake failed for <addr>`, на клиенте — обрыв сразу после `Handshake complete` (`decryption failed`). Самый неочевидный случай |
| **Ключ сервера (пиннинг)** | `qeli show-identity` | `key` / `auth.server_public_key` | `SERVER KEY MISMATCH … possible MITM attack!` — с подсказкой, что делать при ротации |
| **`bind_static` + нулевой ключ** | — | `key = 0000…` (TOFU-заглушка) при `bind_static = true` | Фатально ещё до сети: `bind_static_to_session is on but server_public_key is the all-zero TOFU sentinel` |
| **Транспорт** | `bind.transport` (`tcp`/`udp`) | `proto` | Соединение просто не устанавливается: TCP-клиент стучится в UDP-порт и наоборот |
| **Wire-режим** | `obf.mode` | `mode` | Сервер не распознаёт поток: таймаут хендшейка, в debug-логе `handshake timeout` |
| **`obfs_key` (для `mode = obfs`)** | `obf.obfs_key` | `obfs_key` | Поток расшифровывается в мусор → хендшейк не парсится |
| **REALITY `short_id`** | `obf.tls.reality_proxy.short_ids` | `reality_sid` | Сервер **молча** считает вас чужим и мостит на target: «не подключается, ошибок нет», а `curl` до сервера показывает настоящий сайт |
| **AmneziaWG junk** | `obf.awg.jc` / `jmin` / `jmax` | `awg` / `jc` / `jmin` / `jmax` | Хендшейк не проходит: сервер ждёт другое число junk-пакетов. Включать нужно **на обеих** сторонах с одинаковым `jc` |
| **Часы (для REALITY)** | — | — | Токен несёт timestamp с окном **±120 с**. Разошлись часы → тот же симптом, что и с неверным `short_id`: молча мостит на target. Нужен NTP |
| **Профиль, к которому пиннингуетесь** | у **каждого** профиля свой `identity/<профиль>.key` | `key` | Пин от профиля A против порта профиля B → `SERVER KEY MISMATCH`. Ключ не общий на сервер, а на профиль |
| **`require_client_key_proof`** | `auth.require_client_key_proof` | наличие пина | Обратное направление: сервер отвергает **непиннингованного** клиента — `AUTH DENIED … server key not pinned`, и попытка идёт в счётчик брутфорса по IP |
| **`obf.fronting`** | `obf.obfs_fronting` (по умолчанию `websocket`) | `front` | Для `mode = obfs`: расхождение → `obfs ws: server did not switch protocols` |

**Практика.** Не собирайте клиентский конфиг руками — импортируйте `qeli://`-ссылку
(`qeli add-client <user> --link --host <host>` или кнопка в панели): она несёт `host:port`,
`user`, `pass`, `proto`, `mode`, `key`, `sni`, `reality_sid`, `obfs_key`, `awg`/`jc`/`jmin`/
`jmax` — то есть **все** строки таблицы выше, кроме `bind_static` (он и там, и там по
умолчанию включён) и часов.

Отдельно про роутеры: в шаблоне OpenWrt (`qeli-openwrt/files/qeli.config`) ключ — нулевая
TOFU-заглушка, и `bind_static` под неё выставлен в `'0'`. Как только вы прописываете
**настоящий** ключ, обязательно переведите и `bind_static` в `'1'`, иначе получите строку
1 из таблицы.

## 2. Проверка конфига перед запуском

Отдельной команды «проверить конфиг и выйти» **нет** (на 0.7.11). Валидация происходит
при старте data-plane worker'а, поэтому проверить можно так:

```bash
systemctl stop qeli
qeli server --config /etc/qeli/server.conf
# смотрим первые строки; при ошибке worker падает с понятным сообщением
# (каталог ошибок — TROUBLESHOOTING.md §4.1). Прерываем Ctrl-C.
```

> ⚠️ **Прерывать обязательно вручную.** Супервизор перезапускает упавший worker по кругу
> с нарастающей паузой (`supervisor: worker exited after Ns — respawning in Ns`), так что
> сам процесс не завершится даже на заведомо битом конфиге.

**Чего эта проверка НЕ ловит: опечатки в именах ключей.** Неизвестный ключ не считается
ошибкой и не пишет предупреждение **ни на каком уровне логирования** — он просто не
читается, а параметр молча остаётся дефолтным. Так, например, `exclude_routes` вместо
`exclude` годами выглядел рабочим, а split-tunnel при этом не применялся. Поэтому после
правки конфига **проверяйте результат, а не отсутствие ошибок**: `qeli list-clients`,
маршруты в `ip route`, реальный обход туннеля.

## 3. Обновление и откат

### 3.1. Установка из `.deb` (обычный путь)

В корне репозитория есть [`update-qeli-server.sh`](../../update-qeli-server.sh) — он
делает всё правильное сам:

```bash
sudo ./update-qeli-server.sh
```

Что именно он делает:
1. Определяет текущую версию и последний релиз на GitHub (`QELI_REPO` переопределяет репозиторий).
2. Качает `.deb` **и `SHA256SUMS`** и **сверяет контрольную сумму** — при несовпадении не ставит.
3. **Копирует текущий бинарь** в резервную копию рядом.
4. Ставит пакет и перезапускает `qeli`.
5. Если служба **не поднялась** — **сам возвращает старый бинарь** и перезапускает снова,
   а если и это не помогло, честно сообщает, что служба лежит, и просит смотреть журнал.

То есть откат при неудачном обновлении встроен; отдельных действий не требуется.
`QELI_FORCE=1` переустанавливает даже текущую версию.

Три оговорки:

- **Откат — только бинаря.** Скрипт не бэкапит конфиг, пользователей и identity-ключ и
  не откатывает метаданные dpkg (пакет числится новым, а бинарь на диске старый). Перед
  крупным обновлением снимите бэкап (см. [§4](#4-что-бэкапить)).
- **`QELI_DEB=/путь/к.deb` отключает проверку контрольной суммы** — сверять нечего.
  Осознанно ставьте так только свои сборки.
- **Подписи нет**, только SHA256 из того же релиза. Гарантия целостности, но не
  происхождения.

Ручной эквивалент, если скрипт использовать не хотите:

```bash
sudo cp -a "$(command -v qeli)" /root/qeli.bak      # 1. бэкап бинаря
sudo apt install ./qeli_<версия>_amd64.deb          # 2. установка
sudo systemctl restart qeli && systemctl status qeli
# откат, если не поднялось:
sudo systemctl stop qeli && sudo cp -a /root/qeli.bak "$(command -v qeli)" && sudo systemctl start qeli
```

> **Ваши конфиги пакет не трогает.** `.deb` кладёт только `*.conf.example`; рабочие
> `/etc/qeli/server.conf`, `users.conf` и identity-ключ в пакет вообще не входят —
> перезаписать их нечем. Поэтому «обновление» — это замена бинаря, а не переустановка
> конфигурации.

Три вещи, о которых стоит знать заранее:

- **Не правьте `*.conf.example` на месте.** Они — dpkg-conffiles, и при обновлении apt
  спросит «оставить вашу версию или взять новую». Копируйте (`cp server.conf.example
  server.conf`) и правьте копию — тогда обновления проходят молча.
- **`postinst` делает `chown -R qeli:qeli /etc/qeli` при каждой установке.** Содержимое
  не меняется, но владелец файлов — да. Если вы намеренно выставляли другие права,
  проверьте их после обновления.
- **`apt remove` выполняет `systemctl disable`.** После переустановки автозапуск нужно
  включить снова: `systemctl enable --now qeli`. А `apt purge` **оставляет `/etc/qeli`**
  (в пакете нет `postrm`) — то есть ключи и хеши паролей остаются на диске; удаляйте
  вручную, если этого не хотите.

### 3.2. Docker

Образ не обновляется «на месте» — меняется тег и пересоздаётся контейнер. Состояние живёт
в томе `/etc/qeli`, поэтому конфиги, пользователи и ключ переживают пересоздание:

```bash
docker compose -f release/docker/docker-compose.yml pull
docker compose -f release/docker/docker-compose.yml up -d
```

Откат — вернуть предыдущий тег образа в `docker-compose.yml` и повторить `up -d`.
Подробности и оговорки — [release/docker/README.md](../../release/docker/README.md).

> **`update-qeli-server.sh` для Docker не годится.** Он ищет контейнер с именем ровно
> `qeli`, а compose создаёт `qeli-server` — то есть на compose-развёртывании
> docker-ветка скрипта просто не срабатывает. Но даже когда срабатывает, он делает
> `docker restart` **существующего** контейнера, а тот продолжает работать на старом
> образе. Обновляйтесь через `docker compose pull && up -d`.

> **Клиентская роль в Docker требует `dns = off`.** `/etc/resolv.conf` в контейнере —
> bind-mount, и дефолтный `dns = tunnel` падает с EBUSY и уходит в реконнект-петлю.
> Entrypoint предупреждает об этом при старте.

### 3.3. Совместимость версий при обновлении

Между клиентом и сервером **нет согласования версий** — нет ни минимальной поддерживаемой
версии, ни отказа по номеру. Совместимость определяется исключительно совпадением
параметров из [§1](#1-что-должно-совпадать-у-клиента-и-сервера). Практический вывод:
обновляйте **сервер первым**, клиенты — по мере возможности; ломается не версия, а
изменившийся дефолт (так было с `bind_static` в 0.7.1 и с `handrolled` позже).

Перед обновлением прочитайте раздел своей версии в [CHANGELOG.md](../../CHANGELOG.md) —
смена дефолта всегда отмечена там явно.

## 4. Что бэкапить

Бэкап можно снять из панели (**Backup** → скачать архив) или руками. Минимальный набор,
**без которого восстановление невозможно**:

| Что | Путь по умолчанию | В архиве панели | Почему критично |
|---|---|---|---|
| **Identity-ключ профиля** | `/etc/qeli/identity/<профиль>.key` (0600, создаётся при первом старте) | да | Восстановлению **не подлежит**. Потеряете — сервер молча сгенерирует новый при старте, и **все** клиенты с пиннингом получат `SERVER KEY MISMATCH`. Ключ **у каждого профиля свой** |
| Конфиг сервера | `/etc/qeli/server.conf` | да | Профили, порты, режимы, REALITY `short_ids`, `password_hash` панели |
| Пользователи | `/etc/qeli/users.conf` | да | Логины, argon2-хеши, лимиты, ACL, статические IP |
| TLS-серт и ключ панели | `/etc/qeli/web-tls-{cert,key}.pem` | да | Иначе браузеры снова начнут ругаться на self-signed |
| Ссылки клиентов | `/etc/qeli/client-links/` | да | Готовые `qeli://` и **пароли открытым текстом** — хранить как секрет |
| **Ключ сессий панели** | `/var/lib/qeli/.session_key` | **НЕТ** | Под systemd (`StateDirectory=qeli`) лежит **вне** `/etc/qeli`. Потеря не фатальна — разлогинит всех, — но архив панели его не содержит |
| **TOFU-пины клиента** | `/var/lib/qeli/known_hosts` | **НЕТ** | Только на машинах, где qeli работает клиентом |

> **Архив из панели покрывает только `/etc/qeli`.** Всё, что лежит в `/var/lib/qeli`,
> в него не попадает. Если снимаете бэкап руками — берите оба каталога.

```bash
sudo systemctl stop qeli
sudo tar czf qeli-backup-$(date +%F).tar.gz -C / etc/qeli var/lib/qeli
sudo systemctl start qeli
```

Полезные детали реализации:

- Панель **отказывается** отдать архив, если identity-ключи оказались нечитаемыми (панель
  работает от пользователя `qeli`, а `tar --ignore-failed-read` молча пропустил бы их).
  Вместо битого архива вернётся ошибка с указанием починить права
  (`chown -R qeli:qeli /etc/qeli/identity`) или снимать бэкап от root. Это ровно тот
  случай, когда «бэкап есть, а восстановиться нельзя».
- **Restore сам делает снимок** текущего состояния в `/etc/qeli/.pre-restore-<ts>.tgz`
  (0600, хранятся последние 5) — то есть неудачное восстановление обратимо. Сами эти
  снимки в новый архив не включаются. После restore нужен **ручной перезапуск**.

> Архив содержит **и ключ, и хеши паролей, и пароли клиентов открытым текстом**. Храните
> его как секрет: не в общем облаке, не в репозитории.

**Проверьте восстановление заранее**, а не в момент аварии: разверните архив на запасной
машине, поднимите сервер, подключитесь клиентом с пиннингом. Только это подтверждает, что
identity-ключ действительно сохранён.

## 5. Файрвол: что открывать

Минимум — **порт каждого включённого профиля** (`bind.port`, с правильным протоколом) и,
если панель нужна снаружи, её порт.

| Что | Порт по умолчанию | Протокол | Открывать |
|---|---|---|---|
| VPN-профиль | `443` | по `bind.transport` — **TCP или UDP** | всегда |
| Дополнительные профили | 8443–8450 в multiprofile-шаблоне | TCP/UDP по профилю | если включены |
| Веб-панель | `8080` | TCP | **только если** нужна снаружи |

Важные оговорки:

- **UDP-профиль требует UDP-правила.** Открытый TCP/443 не пропустит `udp-quic` на 443 —
  это разные правила в облачном Security Group.
- **DNS наружу открывать не нужно.** Резолвер qeli работает **внутри** туннеля.
- **Панель лучше не публиковать.** Безопаснее оставить `bind = 127.0.0.1` и ходить через
  SSH-туннель: `ssh -L 8080:127.0.0.1:8080 root@сервер`. Если всё же публикуете —
  обязательны `password_hash` (без него публичный bind просто не стартует) и
  `allowed_ips`. Учтите: `install-reality-server.sh` **включает панель на `0.0.0.0:8080`
  сам** — см. §2 GETTING-STARTED.
- **Правила для туннеля qeli ставит сам**, если у профиля включён `routing.nat.enabled`:
  `ip_forward`, MASQUERADE, `FORWARD … ACCEPT` и MSS-clamp, с тегом `qeli-nat:<профиль>`
  и снятием при чистой остановке. Дублировать их вручную не нужно — подробности и
  единственное исключение см. в [CONFIG.md](CONFIG.md), раздел «Тюнинг ОС сервера».
- **iptables-nft и legacy.** qeli работает **только** через CLI `iptables` (никогда `nft`
  или `ufw`) и проверяет результат не по коду возврата, а перечитывая правила (`-C`) —
  на nft-обёртке код возврата врёт. Правила разделены на обязательные (MASQUERADE и два
  MSS-клампа: при неудаче установка откатывается и профиль не стартует) и
  необязательные (`FORWARD … ACCEPT`: при неудаче только предупреждение). Если увидите
  в логе `FORWARD ACCEPT rules could not be applied (host has a mixed legacy/nft filter
  table)` — егресс всё равно работает при политике FORWARD `ACCEPT`, но при `DROP`
  форвардинг придётся разрешить самому.
- **OpenWrt.** Там свой файрвол (fw4/nftables): пакет **один раз при установке** создаёт
  зону `qeli` и правило `lan → qeli`. Это сделано именно потому, что сырые iptables-правила
  fw4 вычищает при `/etc/init.d/firewall reload`. Побочный эффект: зона привязана к имени
  устройства на момент установки — **смените `qeli.main.dev` потом, и зона останется
  указывать на старое имя**. Подробности — [qeli-openwrt/README.md](../../qeli-openwrt/README.md).

Проверить, что именно поставил qeli:

```bash
sudo iptables-save | grep qeli-nat
```

## 6. Тихие ловушки конфигурации

Эти настройки не вызывают ошибок — они просто не делают того, чего вы от них ждёте.
Сервер предупреждает о них в логе **при старте**, поэтому после правки конфига стоит
прочитать первые строки `journalctl -u qeli`.

| Настройка | Когда бесполезна | Что в логе |
|---|---|---|
| `obf.awg.*` (junk) | на TCP-профиле `fake-tls` / `reality-tls` | `obf.awg.enabled has no effect on a TCP '<mode>' profile` — junk шлётся только на TCP-`obfs` и на любых UDP |
| `obf.multipath.*` | на UDP-транспорте | `has no effect on a UDP transport` — бондинг потоков только для TCP, UDP-сессия ограничена одним потоком |
| `web.secure_cookie` | на панели по HTTP | Cookie с флагом `Secure` браузер по HTTP не отправит — вы просто не сможете войти |
| `web.allowed_origins` не задан | панель за прокси/по домену | Страница грузится, но любой POST — 403 (CSRF по Origin) |
| Секция `[profiles.performance]` целиком удалена | всегда | Serde заполнит её **нулями**, а не дефолтами: `handshake_timeout = 0` и `max_clients = 0` отвергнут всех. Это ловится на старте с явным сообщением |

Отдельно: **опечатка в имени ключа не логируется вообще** — см.
[§2](#2-проверка-конфига-перед-запуском).
