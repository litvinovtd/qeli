# qeli — external audit review and fixes (2026-06-11)

Source: an external audit of qeli v0.6.0 (brought in by the user). Each item is checked
against the code — the format as in [`AUDIT-2026-06-10.md`](AUDIT-2026-06-10.md): an ID, a
severity (claimed by the audit → the real one per the check), the files, the approach, the
acceptance criterion.

Status legend: ✅ made · ⬜ deferred/triage · ❎ a false finding / outdated / by design.

> **The meta-conclusion on the audit.** The inventory is useful, but the severity is
> systematically inflated and there are false positives: the "CRITICAL" T1 isn't critical
> (the proof is ephemeral per-handshake — a byte-by-byte timing brute-force is
> impractical); T3 is a false positive (it's a correct snapshot-under-lock, not a TOCTOU);
> T5/T12 are unreachable/trusted input; B3 "not in VCS" is outdated (the repository is
> under git); T2 "364 panics" is inflated by test code (`#[cfg(test)]`) and
> `lock().unwrap()`. Genuinely deserving of fixes were L1 (DoS), T6–T10 (reliability), and
> T1 (constant-time hygiene).

> **Verification (2026-06-11) — all stacks green.** Rust (.10): `build --release` OK ·
> `test --all` **188 passed / 0 failed** · `clippy -D warnings` 0 · `fmt --check` clean.

## L1 — 🟡 A lockout-DoS by username → replaced with an adaptive tarpit (✅)

**Files:** `qeli/src/server/mod.rs` (`FailedAuthTracker`), `qeli/src/server/handler.rs`
(`verify_client_auth`), `qeli/src/web/api/login.rs`, `qeli/src/web/auth.rs`.

**The problem.** The old `FailedAuthTracker` after `max_attempts` failures **hard-locked
the username itself** (the `by_user` bucket with a `lockout-until`), checked in `check()`
regardless of the source-IP. Knowing a valid login, an attacker from any IPs made 5 failed
attempts and **knocked out the legitimate user** (a classic account-lockout DoS). In the
previous audit (L1, 2026-06-10) it was left "by design" — now it's closed without a
compromise.

**The approach (made) — the "adaptive tarpit" variant.** The hard lock stays **only by
source-IP** (`by_ip`); the username is **never blocked**. Instead of a lock, a username
under an active brute-force gets a **capped exponential delay** (`user_tarpit()`: base
200ms → ×2 per extra failure, a ceiling of 3s). The caller sleeps it before the Argon2
check, so:
- a legit user with the **correct** password is never refused — at worst, during an
  attack, their successful attempt is delayed ≤3s (from **any** IP, including a new one);
- a distributed brute-force (a botnet, IP rotation) is throttled by the aggregate guessing
  rate;
- a single source-IP is still hard-locked by `by_ip`.

API: `check(user,ip)` → `check_ip(ip)` + `user_tarpit(user)`; added
`record_ip_failure(ip)` — the server-key-proof rejection path writes **only** to `by_ip`,
so that a keyless scanner doesn't raise someone else's username-tarpit. Applied uniformly
in the web panel too (the login form + Basic-auth).

**Acceptance:** a login flood from many IPs yields `user_tarpit(...) > 0`, but the
victim's clean IP is always `check_ip(...).is_ok()` — coded by the test
`username_flood_never_hard_blocks_a_clean_ip` (`mod.rs`).

## T1 — ⚪ Android: the auth-proof comparison is not constant-time (✅)

**File:** `qeli-android/.../QeliService.kt` (`verifyServerAuth`).

**The problem.** The server's proof was compared via `ByteArray.contentEquals()`, which
short-circuits on the first mismatched byte → formally a timing oracle. *The "critical"
severity is inflated*: `expected` is derived from ephemeral handshake values
(transcript/ephemeral) and **changes every connection**, so an iterative byte-by-byte
brute-force by timing is impractical. It's fixed cheaply, as hygiene.

**The approach (made).** `MessageDigest.isEqual(receivedProof, expected)` (constant-time).
The pinned-key comparison (`serverStaticPub.contentEquals(pinnedBytes)`) is left as is — it's
a public key, not a secret. Rust (`ct_eq`) and C# (`FixedTimeEquals`) were already
constant-time — the finding is purely Android.

## T6 — 🟡 A poisoned lock tears down a live session (✅)

**File:** `qeli/src/server/handler.rs` (`SessionShared`).

**The problem.** `stream_count()` (`unwrap_or(0)`) and `remove_stream()` (`Err(_) => true`)
on a poisoned `streams` mutex reported "0 streams" / "the session is empty" → a false
teardown of a live session.

**The approach (made).** All `SessionShared` methods (`pick_stream`, `kick_all`,
`remove_stream`, `stream_count`, and the new `try_add_stream`) switched to the existing
helper `lock_or_recover` — it recovers the guard from poison instead of a silent
degradation.

## T7 — 🟡 TOCTOU: exceeding `max_clients` (✅)

**File:** `qeli/src/server/handler.rs` (the AUTH branch).

**The problem.** The limit was checked under a read-lock, while the session insertion —
separately under a write-lock; N concurrent connects could all pass the check and exceed
the limit.

**The approach (made).** An authoritative re-check `by_ip.len() >= max_clients` **under the
same write-lock** as the insertion; on exceeding — releasing the already-issued IP
(`pool.release(&dkey)`) and a refusal. The early read-check is left as a cheap fast-path
(not to allocate an IP in the typical over-limit case).

---

## T8 — 🟡 TOCTOU: exceeding `max_streams` (✅)

**File:** `qeli/src/server/handler.rs` (`SessionShared`, `run_stream`).

**The problem.** A JOIN checked `stream_count() >= max_streams`, while the actual
`add_stream` went later as a separate lock acquisition — parallel JOINs slipped past the
limit.

**The approach (made).** An atomic `try_add_stream()` — the length check and the push under
one lock; returns `false` when full. In `run_stream` the stream that lost the race is
dropped (a log + return) instead of exceeding the cap. The early check in JOIN is left as a
fast-path (a quick refusal before sending `JOINOK`).

---

## T9 — ⚪ A `DeviceId()` race (✅)

**File:** `qeli-shared/.../VpnTunnelBase.cs`.

**The problem.** Without a file lock, two concurrent calls (the primary + the bonded
streams at start) could generate and write two different device-ids.

**The approach (made).** A static cache `_deviceId` + `lock (_deviceIdLock)` — the id is
computed once per process, read/generate/write are serialized.

---

## T10 — 🟡 A socket leak on a connect/handshake error (✅)

**Files:** `qeli-shared/.../VpnTunnelBase.cs` (`OpenBondedStream`),
`qeli-android/.../QeliService.kt` (`openBondedStream`).

**The problem.** A socket was created, and if `connect`/the JOIN handshake threw **before**
it was added to the teardown list — it wasn't closed (an FD leak on a flapping network,
especially with repeated bonded attempts).

**The approach (made).** The body is wrapped in a `try/catch`: on any exception the socket
is closed (`Close()`/`ch.close()`) and removed from `_bondedSockets`/`bondedSockets` if it
was already added. Then the exception is rethrown.

## Deferred / triage

- **T2** (✅ 2026-06-19) Triaged panics on incoming-byte parsing paths. Walked all of
  `protocol/` + `server/handler.rs` + `udp_handler.rs`: the parsers are defensively
  written (length guards before every index, `.get()?`, clamped ext-end, loop bounds in
  `clienthello`, `tls`, `realtls/{server,record,sansio}`, `quic`, `obfs`). **One real
  bug:** `packet.rs::decrypt_packet` — `payload[..NONCE_SIZE]` panicked when
  `payload_len < NONCE_SIZE` (the length field is attacker-controlled, bounded only
  above). Unreachable over TCP (`record.len()==header+payload_len`), but over **UDP** a
  short datagram cleared the checks → panic → under `panic="abort"` a **pre-auth remote
  crash-DoS**. Fixed with `payload.get(..NONCE_SIZE).ok_or(..)` + a regression test
  (raw+tls framing). The remaining `unwrap/expect` are tests, `lock().unwrap()`, and
  own-data invariants (fixed-length keys etc.) — not wire data.
- **T11** (✅ 2026-06-19) `Thread.sleep` in the Android IO readers. The channels are
  blocking (`configureBlocking(true)`) → `read()` returns ≥1 or −1, so the `n==0`+`sleep`+
  retry-as-timeout branch was a dead busy-wait (and not a real timeout). Removed from
  `Stream.readRaw`/`readSomeRaw` (`QeliService.kt`); real liveness is the `rxDead` deadline
  in the data-plane/heartbeat loops. `soTimeout` does not apply to NIO `SocketChannel.read()`,
  hence the dead-code removal. APK 702/0.7.2 rebuilt.

## False findings / no action

- **T4** (❎ by design) disable-user sets `enabled=false` in memory before `save`. This is
  fail-safe: the user is disabled in the live process even on a write failure, and
  `save_err` is returned to the admin. The order is kept.
- **T3** (❎ false) config.rs "lock-drop-then-use TOCTOU". The path is cloned under the lock
  (`p.clone()`), then `drop` — this is a correct snapshot, the write goes by the local
  copy; there's no check-then-use over shared state.
- **T5** (❎ unreachable) `build_auth_ok` serializes a `serde_json::json!` `Value`, which
  practically never fails; `unwrap_or_default()` is a dead branch.
- **T12** (❎ trusted input) `build_routes_json` formats `cidr/gateway/metric` from the
  **admin server configuration**, not from an attacker's network input. Not an injection.
- **B3** (❎ outdated) "the release is not in VCS" — the repository is under git (a commit
  history, `release/` in the index).
