## D1 â€” đź”´ The server doesn't write to `/etc/qeli` under the stock systemd unit

**Files:** `qeli/debian/qeli.service`, `qeli/src/server/mod.rs` (`generate_profile_key`),
`qeli/src/config/server.rs` (`default_users_file`), `qeli/src/server/control.rs`.

**The problem.** `ProtectSystem=full` mounts `/etc` read-only, but the server writes there
by default: (1) generating the identity key `/etc/qeli/identity/<name>.key` at first start
â†’ `EROFS` â†’ the profile isn't bound; (2) `users.save()` from the panel and control
commands (`disable-user`/`enable-user`/`set-bandwidth`) â†’ `EROFS`, and moreover
`control.rs` logged the error but returned `ok:true` ("a silent success").

**The approach (made).** `ReadWritePaths=/etc/qeli` added to the unit â€” the key paths were
**not** changed (the production identity key isn't regenerated, the client pinning is
intact), the rest of the FS stays read-only. `control.rs`: the three commands now return
`ok:false` with clear text if saving the users file failed.

**Acceptance:** on a clean machine under the unit `systemctl start qeli` brings up the
profile and generates the key; `qeli disable-user X` with a non-writable users file
returns an error, not `OK`.

## W2 â€” đźź  The web panel's Argon2 blocked the async runtime

**Files:** `qeli/src/web/auth.rs`, `qeli/src/web/api/login.rs`, `qeli/src/web/pages/*.rs`.

**The problem.** `verify_credentials`/`is_authed`/`check_auth` ran Argon2 synchronously
right in the async handler (on every `POST /api/login` and on every API request with HTTP
Basic). A multi-threaded tokio runtime: several concurrent requests clog all workers with
Argon2 work â†’ the supervisor stalls (the panel + the control-socket relay). The VPN path
was done right (`handler.rs` â€” `spawn_blocking`).

**The approach (made).** `is_authed`/`check_auth`/`verify_credentials` became `async`;
Argon2 moved into `spawn_blocking`. The cheap cookie check (HMAC) stayed inline. The Basic
parsing moved into a synchronous `basic_credentials` (without Argon2). All callers
(`AuthGuard` + 5 pages + login) switched to `.await`.

**Acceptance:** a surge of requests to `/api/login` doesn't hang the panel; the cookie
path without an extra thread-hop.

## W1 â€” đźź  No brute-force protection on the panel login

**Files:** `qeli/src/web/api/login.rs`, `qeli/src/web/mod.rs`.

**The problem.** `login` didn't use the `FailedAuthTracker` (unlike the VPN-auth). The
CSRF middleware doesn't prevent a targeted brute-force via curl.

**The approach (made).** Login reuses the existing `failed_auth` (in the supervisor's
`ServerState`), the key â€” username + the source-IP via `ConnectInfo`. On the limit â€” a
`429`. The tracker's locks aren't held for the duration of Argon2. In `web/mod.rs`
`into_make_service_with_connect_info::<SocketAddr>()` is enabled.

**Acceptance:** N failed logins (default `brute_force`: 5/15 min) â†’ a `429`.
**Note:** only `/api/login` is covered; the Basic-auth path â€” see **W1b**.

## U2 â€” đźźˇ A UDP state flood via source-IP spoofing (âś…)

**Files:** `qeli/src/server/udp_handler.rs`.

The rate-limit is only for new sessions and by `addr.ip()`; by spoofing random IPs one can
breed `AwaitingAuth` records (cleaned by `handshake_timeout`). **Made:** the constant
`MAX_PENDING_HANDSHAKES = 1024` (per worker); on reaching the cap, before inserting a new
half-open session, the oldest `AwaitingAuth` is evicted (the authenticated ones are not
affected). **It doesn't close U1** (amplification) â€” only the memory.

## C1 â€” đźźˇ Win: the service credentials are locally recoverable (âś…)

**Files:** `qeli-win/QeliWin/Service/ServiceState.cs` (`SaveProfile`/`RestrictProfileAcl`).

The profile (password+obfs_key) is encrypted with DPAPI `LocalMachine` (decryptable by any

have read). **Made:** the ACL is set on the profile **file** itself in `SaveProfile` (not

removed. Everything in a `try/catch` â€” an ACL failure doesn't break saving (the DPAPI
encryption stays). The build confirmed: `FileInfo.SetAccessControl`/`FileSecurity`/
`WellKnownSidType` resolve on net10-windows. **A manual e2e on Windows remains:** writing
the profile from the GUI (user) + reading it by the service (SYSTEM) under the new ACL.
